1. Разобрал коллекцию по группам API:
   - `api/1`: объявления, статистика, объявления по продавцу.
   - `api/2`: удаление объявления, статистика.
2. Для каждого endpoint выделил позитивные и негативные сценарии по кодам ответа, которые заданы в коллекции (`200/400/404/500`, а также `100` для одного запроса).
3. Добавил унифицированный шаблон кейсов: **ID, приоритет, шаги, ожидаемый результат**.
4. Сфокусировался на том, чтобы кейсы были пригодны для ручного прогона и легко переводились в автотесты.

## Общие предусловия

- Базовый URL: `https://qa-internship.avito.com`
- Заголовок `Accept: application/json` для всех GET/DELETE.
- Для POST: `Content-Type: application/json`, `Accept: application/json`.
- Есть валидные тестовые данные:
  - существующий `id` объявления;
  - несуществующий `id` объявления;
  - валидный `sellerID`.

## Допущения

- Если коллекция показывает код в примере ответа, он считается целевым ожидаемым кодом.
- Для `500` кейсов проверяется только корректность обработки серверной ошибки (структура ответа), а не бизнес-логика.
- Код `100 Continue` оставлен как отдельный кейс по коллекции, хотя в реальном REST API обычно не используется в бизнес-сценариях.

## API v1

### 1) GET `/api/1/item/:id` — Получить объявление по идентификатору

### TC-API1-ITEM-GET-001 (High)
- **Сценарий:** Успешное получение объявления по валидному `id`
- **Шаги:**
  1. Отправить `GET /api/1/item/{valid_id}`.
- **Ожидаемый результат:**
  - HTTP `200`.
  - Ответ `application/json`.
  - Тело — массив объектов объявления с полями: `id`, `sellerId`, `name`, `price`, `statistics.likes`, `statistics.viewCount`, `statistics.contacts`, `createdAt`.

### TC-API1-ITEM-GET-002 (High)
- **Сценарий:** Некорректный формат `id`
- **Шаги:**
  1. Отправить `GET /api/1/item/{invalid_format_id}`.
- **Ожидаемый результат:**
  - HTTP `400`.
  - Тело содержит `result.messages`, `result.message`, `status`.

### TC-API1-ITEM-GET-003 (High)
- **Сценарий:** `id` не найден
- **Шаги:**
  1. Отправить `GET /api/1/item/{not_existing_id}`.
- **Ожидаемый результат:**
  - HTTP `404`.
  - Тело содержит `result` и `status`.

### TC-API1-ITEM-GET-004 (Medium)
- **Сценарий:** Внутренняя ошибка сервера
- **Шаги:**
  1. Смоделировать/воспроизвести условие для серверной ошибки.
  2. Отправить `GET /api/1/item/{id}`.
- **Ожидаемый результат:**
  - HTTP `500`.
  - Тело содержит `result.messages`, `result.message`, `status`.

## Corner cases (граничные и нестандартные сценарии)

Ниже — кейсы вне «счастливого пути»: границы типов, кодировка URL, дублирующие запросы, заголовки и несогласованность полей в теле (как в реальном стенде).

### Общие (все методы)

| ID | Сценарий | Действия | Ожидаемое поведение (зафиксировать фактическое) |
|----|----------|----------|--------------------------------------------------|
| TC-CORNER-001 | Нет заголовка `Accept` | Отправить запрос без `Accept` | Либо `200` + JSON, либо предсказуемая ошибка; не должно быть необработанного исключения на сервере. |
| TC-CORNER-002 | `Accept: */*` | Указать `Accept: */*` | Ответ парсится как JSON при успехе. |
| TC-CORNER-003 | Дублирующиеся заголовки | Дважды `Accept` с разными значениями | Поведение по RFC (обычно берётся первое/последнее) — задокументировать. |

### GET `/api/1/item/:id` и GET `/api/2/statistic/:id` (по аналогии для `:id` в пути)

| ID | Сценарий | Действия | Ожидаемое |
|----|----------|----------|-----------|
| TC-CORNER-010 | Пустой сегмент пути | `GET .../item/` или пропуск `id` (если клиент позволяет) | `404` или `400`, не `500` без тела. |
| TC-CORNER-011 | UUID в другом регистре | Валидный UUID, но буквы в `A-F` в верхнем регистре | Если сервер нормализует — `200`; иначе зафиксировать факт. |
| TC-CORNER-012 | Пробелы вокруг `id` | URL-encoded пробелы в конце `id` | `400` или `404`, не «тихий» успех с чужими данными. |
| TC-CORNER-013 | Очень длинная строка вместо `id` | Строка > 1–4 КБ в path | `400`/`414`/`404` — без падения процесса. |
| TC-CORNER-014 | SQL/скрипт в `id` | `id` вида `'; DROP--` | Ответ только JSON/ошибка, без утечки стека в теле. |

### POST `/api/1/item` — Сохранить объявление

| ID | Сценарий | Действия | Ожидаемое |
|----|----------|----------|-----------|
| TC-CORNER-020 | Пустое тело | `POST` с `Content-Length: 0` | `400` или явный `415`; не «успех» с мусором. |
| TC-CORNER-021 | Не JSON | `Content-Type: text/plain`, тело `{...}` | `400`/`415`. |
| TC-CORNER-022 | Невалидный JSON | Обрезанная скобка, лишняя запятая | `400` с понятной ошибкой. |
| TC-CORNER-023 | `sellerId` vs `sellerID` | Тело только с `sellerID` (как в Postman) и только с `sellerId` | Оба варианта: зафиксировать, какой ключ реально принимает бэкенд. |
| TC-CORNER-024 | `statistics.like` vs `statistics.likes` | Как в старом API-тесте — поле `like` вместо `likes` | Либо принимается с алиасом, либо `400` с полем ошибки. |
| TC-CORNER-025 | Границы `price` | `0`, `-1`, `MAX_INT`, `1e20` как число | Согласованная валидация (не сохранение «тихо» с неверной ценой). |
| TC-CORNER-026 | Пустая строка в `name` | `"name": ""` | Либо отказ `400`, либо явное правило продукта. |
| TC-CORNER-027 | Unicode и длина `name` | Имя 1 символ / 1000+ символов / emoji | `200` с обрезкой или `400` — единообразно. |
| TC-CORNER-028 | Лишние поля в JSON | Неизвестные ключи верхнего уровня | Игнор или `400` — зафиксировать контракт. |

## Рекомендации для дальнейшей автоматизации

- Добавить привязку кейсов к автотестам (`TC-ID` в названии `pytest`-теста).
- Для негативных кейсов стабилизировать тестовые данные (генерация несуществующих `id`).
- Разделить smoke и regression:
  - **Smoke:** все `200` сценарии.
  - **Regression:** `400/404/500` сценарии.
